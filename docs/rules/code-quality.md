# 硬规则 — 代码质量

- **LLM 输出防御**（DEV-BUG-16）：接收 LLM 输出的 Pydantic model 必须加 validator/coerce/strip
- **分支矩阵全覆盖**（DEV-BUG-17）：含分支逻辑必须列矩阵逐格实现和测试
- **流式状态机边界检查**（DEV-71）：写/改流式处理逻辑时，必须逐项验证：①操作顺序 ②标签/定界符被 chunk 边界拆开 ③流结束时缓冲区 flush ④空 chunk / 单字符 chunk ⑤所有下游共用同一份 sanitize 函数。缺任一项 = 未完成
- **改前 grep**（DEV-6）：改代码前 grep 全量引用 + grep 同类 pattern 复用
- **重数据不经渲染进程**（DEV-86）：大体积数据禁止经过渲染进程中转，必须在主进程/后端完成采集和传输
- **修复不能为了改动小而降低质量**（DEV-102）：该改架构就改架构，改动大小不是目标，正确性和可维护性才是
- **第三方库先验底层实现**（DEV-88）：采用前必须先查看平台相关源码，确认底层机制和副作用
- **新需求先横向对比**（DEV-80）：新功能开发前主动搜 2~3 个同类开源项目的实现，确认方向正确后再动手，不要等用户提醒
- **副作用时序**（DEV-103）：先预检条件 → 调用外部服务 → 成功后才执行副作用（扣费/计数/状态变更），禁止先执行副作用再调外部
- **外部资源必须 context manager**（DEV-98）：凡 new 外部 client 默认 context manager；外部 API 返回值必做空值防御
- **代码搬迁必查隐式依赖**（DEV-104）：代码从上下文 A 搬到上下文 B 时，逐项检查：①引用对象生命周期 ②闭包捕获变量有效性 ③初始化时序 ④错误传播路径
