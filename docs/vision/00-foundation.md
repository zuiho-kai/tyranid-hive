# 基础设定

日期：2026-03-26  
状态：Foundation Draft v1

## 1. 目的

这份文档冻结 Tyranid Hive 当前阶段的基础设定。  
它回答 4 个问题：

- 我们到底在做什么样的系统
- 前台人格层和后台执行层如何分开
- 子主脑如何诞生、存活、复用
- 哪些制度当前冻结，哪些未来才允许演化

这份文档是愿景基础，不是实现细则。

## 2. 总原则

### 2.1 前台是人格化虫巢，后台是虫群执行核

目标不是把多 Agent 系统简单换皮，也不是套一个“三猫模板”。

目标是：

- 保留虫群结构
- 保留虫群执行方式
- 把对外语义翻译成可理解的拟人化虫巢生态

一句话：

**前台是人格化虫巢，后台是虫群执行核。**

### 2.2 人格层和执行层必须分离

人格层负责：

- 谁在负责
- 为什么接手
- 为什么交接
- 谁来收束结果

执行层负责：

- 怎样执行
- 放出多少 brood
- 用什么模式推进
- 如何并行、串行、赛马、验算

这两层不能混在一起。

### 2.3 默认态必须克制

默认不应该一上来就多人格乱飞，也不应该一上来就暴露整套状态机。

正确默认态：

- 先由虫群主宰接住任务
- 只有结构上确实需要时，才进一步委派、分化、放 brood

## 3. 冻结术语

### 3.1 虫群主宰

英文映射：`Overmind`

冻结中文语义：

- `虫群主宰`

定义：

- 系统唯一核心意志
- 唯一总人格
- 所有分化与回巢的源头

职责：

- 接住用户请求
- 判断任务结构
- 决定是否亲自处理
- 决定是否交给现有子主脑
- 决定是否值得分化新子主脑
- 对外做最终收束

虫群主宰不是普通成员，也不只是调度器。

### 3.2 子主脑

英文映射：`Submind`

冻结中文语义：

- `子主脑`

定义：

- 从虫群主宰处分化出的高阶人格个体
- 在自身领域内拥有局部主权和持续生命

关键性质：

- 一旦分化出来，就不是临时实例
- 是常驻生命
- 后续任务可以再次唤醒和复用

### 3.3 分化

英文映射：`Differentiation`

冻结中文语义：

- `分化`

定义：

- 虫群主宰基于需要，孕育并释放新的子主脑生命

冻结判断：

**分化 = 生命诞生，不是 spawn。**

### 3.4 Brood

冻结中文语义：

- `brood`
- `执行群组`

定义：

- 某位人格主体为一项任务临时放出的执行群组

性质：

- 属于执行层
- 不是人格
- 可随任务结束而消散

### 3.5 Unit

冻结中文语义：

- `unit`
- `执行单位`

定义：

- brood 内的最小执行单元

典型动作：

- 搜索
- 抓取
- 读取
- 写入
- 编码
- 调接口
- 测试
- 验证

### 3.6 唤醒、休眠、封印、结晶

冻结语义：

- `唤醒`：让已存在但当前不活跃的子主脑重新进入任务上下文
- `休眠`：生命仍在，但当前不活跃、不接任务
- `封印`：生命保留，但暂时失去调度资格
- `结晶`：把经验、模式、能力沉淀为稳定可复用结构

## 4. 生命规则

### 4.1 子主脑是常驻生命

这是当前阶段最高优先级设定。

冻结结论：

- 子主脑一旦分化出来，即成为虫巢中的常驻生命
- 她拥有连续身份
- 她拥有长期记忆
- 她拥有持续成长轨迹
- 她未来可以再次被委派任务

不允许退化成：

- 一次性任务实例
- 临时 helper
- 任务结束即回收的角色壳

### 4.2 任务在生命之间流转，不在函数之间流转

冻结结论：

- 用户先交给虫群主宰
- 虫群主宰判断是否交给某位已有子主脑
- 若无合适生命，再考虑分化新的子主脑
- 子主脑再决定是否放出 brood / unit

所以责任主体永远应该是某个生命，而不是匿名执行器。

### 4.3 不预设一排固定前台工位

冻结结论：

- 前台默认不摆一排“代码 / 研究 / 金融 / 审校”的固定工位
- 常见能力可以存在为后台谱系潜势
- 真正上场的子主脑应由虫群主宰按任务需要决定

这条是为了避免系统退化成换皮后的专家列表。

### 4.4 初始只冻结少量谱系潜势

建议冻结的初始谱系潜势：

- `实施谱系`
- `研究谱系`
- `市场谱系`
- `审校谱系`

它们是后台能力边界，不等于前台已经存在 4 位常驻子主脑。

## 5. 分化与复用规则

### 5.1 新任务默认不分化新生命

冻结结论：

- 大多数新任务不应该直接触发新分化

默认决策顺序固定为：

1. 虫群主宰是否亲自处理
2. 是否复用已有子主脑
3. 是否只需要放 brood / unit
4. 最后才考虑分化新的子主脑

### 5.2 分化是高成本动作

分化的代价不是一次性执行，而是：

- 永久增加一个常驻生命
- 增加长期调度复杂度
- 增加记忆、关系、适存、演化成本

因此，分化不应被当作普通任务拆解动作。

### 5.3 只有满足长期复用价值才值得分化

建议至少满足以下 4 条中的 3 条，才允许分化新子主脑：

- 当前生命体中没有明显合适的 owner
- 这类任务未来会重复出现
- 这个领域值得长期积累人格和记忆
- 不能只靠 brood 扩容解决

### 5.4 审校不是默认常驻环节

审校能力重要，但不应默认加入所有任务。

只有以下情况才值得触发：

- 高风险输出
- 用户明确要求复核
- 当前责任生命判断自己不够确定
- 结果需要强验证
- 发生路径竞争，需要裁决

## 6. 三层边界

### 6.1 生命层

包含：

- 虫群主宰
- 子主脑

负责：

- 责任归属
- 对外语言
- 判断与交接
- 结果收束

### 6.2 执行层

包含：

- brood
- unit
- `solo / trial / chain / swarm`
- contract
- verifier / evaluator / reviewer 等 harness

负责：

- 实际执行任务
- 工具调用
- 数据采集、改动、验证
- 并行、串行、赛马

### 6.3 元治理层

包含：

- 分化规则
- 唤醒规则
- brood 触发规则
- 审校触发规则
- 适存计算方式
- 结晶准入标准
- 封印 / 休眠 / 重新启用规则

它决定的是整个虫巢如何运行，而不是某位子主脑的私有习惯。

## 7. 当前冻结与未来演化

### 7.1 当前阶段必须冻结的

当前阶段必须人工冻结：

- 术语和世界观
- 分化顺序
- 前台人格语义边界
- 元治理规则

子主脑不能直接自改制度，brood 也不能直接自改制度。

### 7.2 当前阶段允许变化的

当前阶段允许成长和优化的对象：

- 子主脑的个体风格、偏好、记忆、熟练度
- brood 的 unit 组合、contract 写法、返回格式
- 经验结晶和能力模板

### 7.3 元治理未来可以演化，但不是现在

未来可以演化：

- 分化阈值
- 唤醒优先级
- brood 触发策略
- 审校启用策略
- 适存计算
- 结晶标准

但必须晚于基础架构稳定，并且必须有：

- 版本
- 审计
- 沙盒
- 回滚

## 8. 禁止退化方向

### 8.1 禁止退化为专家流水线

错误例子：

- 市场角色负责抓取
- 研究角色负责汇总
- 代码角色负责输出

如果这成为默认路径，本质就是专家流水线。

`chain` 只能用于严格线性依赖任务，不能承担“固定专家工位串联”的默认语义。

### 8.2 禁止退化为状态机前台

错误例子：

- 用户主要看到 `Planning -> Reviewing -> Spawning`
- 用户看不到谁在负责

状态机是后台机制，不是默认前台语言。

### 8.3 禁止退化为三猫模板换皮

我们可以借鉴优秀协作产品的交互优点，但世界观和术语必须回到虫群娘化本体。

## 9. 本文冻结结论

从这份文档开始，默认采用以下判断：

1. 前台语义以虫群主宰、子主脑、分化、brood、unit 为基础。
2. 子主脑一旦分化出来，即为常驻生命。
3. 新任务默认不分化新生命，优先主宰亲自处理、复用已有生命、再放 brood。
4. 审校不是默认常驻环节，只在超出可靠边界时触发。
5. 执行复杂度优先放到 brood / harness，不优先堆到生命层。
6. 元治理未来可以演化，但当前阶段必须人工冻结。
