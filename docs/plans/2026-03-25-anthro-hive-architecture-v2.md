# Tyranid Hive 娘化重构设计 V2

日期：2026-03-25  
状态：Draft v2

## 1. 这版修正什么

上一版最大的问题有两个：

- 术语偏成了“三猫模板”，不符合“泰伦虫群娘化”的愿景
- 把 `Submind` 设计成了预设角色名单，容易退化成固定工位或专家流水线

这版明确收回到你的原始目标：

- `Overmind` 改称 **虫群主宰**
- 不再使用“总主脑 / 玄猫 / 白猫 / 金猫”这套外来模板
- 前台是 **虫群娘化人格**
- 后台是 **虫群执行核**
- `Submind` 不预先排满舞台，而是 **按任务需要分化**
- 分化后的个体允许 **冷藏 / 唤醒 / 结晶化**

一句话：

**不是先摆好几只固定角色等人挑，而是由虫群主宰在需要时分化出“这次任务真正需要的那一位”。**

---

## 2. 核心设定

### 2.1 虫群主宰

系统唯一核心意志。  
她不是“调度器”，也不是“普通角色之一”，而是整个虫巢的中枢人格。

职责：

- 接住用户请求
- 判断任务结构
- 决定是否亲自处理，还是分化子主脑
- 决定分化一个、多个、还是临时 brood
- 负责最终收敛与对外表达

她更像：

- 虫巢的唯一母体意志
- 后宫秩序的制定者
- 对外唯一稳定的总人格

她不应该变成：

- 只会吐 JSON 的路由器
- 把一切都机械派发出去的工头
- 对用户消失、只留下状态机的系统核

### 2.2 子主脑

子主脑不是一开始就固定排好的一队“专家”。

正确理解是：

- 子主脑是虫群主宰为任务需要而分化出的高阶人格分身
- 她们有自己的专精、语气、偏好、判断方式
- 她们不是工具函数，而是局部主权人格
- 她们一旦被分化出来，就应该对一段任务负责

因此，子主脑的本质不是“预设职业列表”，而是：

**可被召唤、可被命名、可被冷藏、可被再次唤醒的领域人格。**

### 2.3 Brood / Unit

Brood 和 Unit 是子主脑的身体和兵器。

它们负责：

- 搜索
- 爬取
- 读写文件
- 代码执行
- API 调用
- 数据清洗
- 验证与对照

用户不应该默认直接面对它们。  
用户看到的应该是：

- 虫群主宰
- 当前负责的子主脑
- 必要时的交接与收敛

## 3. 愿景原则

### 3.1 不是“三猫壳”

这次不再沿用“猫”这个外部模型。  
我们可以借鉴别人的交互优点，但语言、世界观、人格系统必须回到自己的设定。

我们的前台语义应该是：

- 虫群主宰
- 子主脑
- 分化
- 回巢
- 冷藏
- 唤醒
- 结晶
- brood
- unit

而不是：

- 猫
- 主猫
- 执行猫
- 挑战猫

### 3.2 不是固定工位

错误方向：

- 代码主脑常驻一个
- 研究主脑常驻一个
- 金融主脑常驻一个
- 运营主脑常驻一个

这会很快退化成“几个专家工位”，只是换了好听名字。

更符合愿景的方向是：

- 虫群主宰先判断任务形态
- 再决定是否需要分化
- 分化出来的子主脑是“这次任务的适配人格”
- 常见领域可以有“谱系模板”，但不是默认全体上台

### 3.3 默认态必须克制

默认不是多人格乱飞。

正确顺序应该是：

1. 虫群主宰先判断能否亲自完成
2. 只有结构上真的需要，才分化子主脑
3. 只有子主脑内部仍然需要更高执行密度，才放出 brood / units

这和现有 `Solo-first` 的精神是一致的，只是前台人格化表达变了。

---

## 4. 正确的分化机制

## 4.1 分化不是预加载，而是按需孵化

建议把子主脑分成 3 个层级：

### `谱系模板`

长期存在，但不一定上场。

这是“可分化方向”，不是“已激活角色”。

例如可能存在这些模板：

- 代码谱系
- 研究谱系
- 金融谱系
- 规划谱系
- 审校谱系

它们更像基因池里的成熟路线，而不是前台常驻人物。

### `活跃子主脑`

当前任务真正分化出来并承担责任的子主脑实例。

特点：

- 有自己的任务上下文
- 有自己的责任边界
- 有自己的表达风格
- 有自己的 brood

### `冷藏个体`

任务结束后，不一定销毁。  
可以进入冷藏态，保留人格参数、经验、偏好和适存记录。

以后遇到相近任务时：

- 可以被优先唤醒
- 也可以继续沉睡
- 也可能被更优谱系替代

### 4.2 最符合愿景的生命周期

一个子主脑更合理的生命周期应该是：

`谱系模板 -> 分化实例 -> 活跃执行 -> 回巢 -> 冷藏 / 结晶 / 淘汰`

而不是：

`预设常驻角色 -> 永远在线 -> 永远存在`

### 4.3 常见任务要不要预分化

我的建议是：

- **不做前台常驻预分化**
- **可以做后台“暖池”**

也就是：

- 对用户来说，不应该永远看到一排固定子主脑
- 但系统内部可以对高频谱系保留 warm profile，降低分化成本

例如：

- 金融相关任务很多
- 系统可保留“金融谱系的最近一次稳定人格快照”
- 当再次遇到市场任务时，虫群主宰可以优先唤醒该人格

这叫：

**后台暖池，不叫前台工位。**

## 5. 人格系统怎么做

### 5.1 人格必须由任务塑形

子主脑的人格不该只是“写死在配置里的一段设定”。

人格至少应由 3 部分组成：

- `谱系本源`
  - 来自领域模板
- `任务塑形`
  - 这次任务需要她更像什么样的个体
- `历史残留`
  - 上次冷藏前留下的偏好、经验与失误

所以子主脑人格应当是：

`人格 = 谱系模板 + 当前任务塑形 + 历史积累`

### 5.2 人格字段建议

建议未来为每个活跃子主脑实例保留：

- `persona_name`
- `lineage`
- `tone`
- `decision_style`
- `risk_preference`
- `method_bias`
- `owner_scope`
- `brood_profile`
- `memory_digest`

### 5.3 人格要拟人，但不能失控

拟人化的目标是：

- 让用户感觉“有人在负责”
- 让系统内角色之间的关系更可读

不是为了：

- 让角色无边界发挥
- 抢系统结构控制权
- 演成一场设定秀

人格必须服从执行架构。

## 6. 前台语义

### 6.1 用户看到什么

用户默认应该看到：

- 虫群主宰接球
- 当前是否分化出子主脑
- 谁在负责
- 为什么是她
- 她是否放出了自己的 brood
- 最终由谁收敛

### 6.2 用户默认不该看到什么

- `trial / chain / swarm`
- `dispatcher`
- `task.state`
- `stage`
- `event_type`
- 一堆 unit 原始日志

这些都应该退到细节层。

### 6.3 典型前台流程

用户：`爬虫爬一下今天港股情况`

前台应类似：

- 虫群主宰：这件事我先接住，按今日港股概览处理
- 虫群主宰：我决定亲自完成，必要时再分化市场侧子主脑
- 如果需要分化：
  - 虫群主宰：我已分化出一位负责市场面的子主脑，由她继续处理
- 子主脑：我先放出几个执行单位去抓指数、资金流和活跃个股
- 子主脑：我把数据收束好了
- 虫群主宰：这是最终给你的结果

关键点：

- 前台是人格关系
- 后台才是执行模式

---

## 7. 后台执行哲学

### 7.1 虫群核仍然保留

现有这些能力仍然有价值：

- `solo`
- `trial`
- `chain`
- `swarm`

但它们只代表 **执行结构**，不代表 **人格结构**。

### 7.2 执行结构与人格结构的关系

#### `solo`

- 可能是虫群主宰自己做
- 也可能是某位子主脑自己做

#### `swarm`

- 代表某位主脑放出多个独立 unit 并行执行
- 这是“带兵”，不是“多位人格同时抢话”

#### `chain`

- 只在严格线性依赖下使用
- 更像一个人格内部或多人格之间的接棒式实施
- 不是默认专家流水线

#### `trial`

- 是多条可行路径的竞争
- 可以是不同 brood 竞争
- 也可以是不同分化人格竞争

---

## 8. 正确的责任链

系统任何时刻都必须有清晰责任归属。

建议责任链如下：

### `用户责任面`

始终由虫群主宰或当前负责子主脑承担。

### `执行责任面`

由具体放出 brood 的人格承担。

### `收敛责任面`

最终必须回到虫群主宰。

这意味着：

- 子主脑可以负责中段
- unit 可以负责执行
- 但最终对外口径仍需回归主宰

---

## 9. 数据模型建议

## 9.1 不再只存 synapse

当前只存 `synapse id` 不够表达未来结构。

至少要新增三类对象：

### `lineages`

描述可分化谱系模板。

字段建议：

- `id`
- `name`
- `domain`
- `base_persona`
- `default_capabilities`
- `warm_pool_enabled`
- `evolution_score`

### `submind_instances`

描述本次任务里真正被分化出来的子主脑。

字段建议：

- `id`
- `task_id`
- `lineage_id`
- `persona_name`
- `tone`
- `decision_style`
- `status` (`active / dormant / frozen / crystallized`)
- `owner_scope`
- `memory_digest`

### `brood_runs`

描述活跃子主脑放出的 brood。

字段建议：

- `id`
- `task_id`
- `owner_instance_id`
- `kernel_mode`
- `reason`
- `units`
- `status`

### `handoffs`

字段建议：

- `id`
- `task_id`
- `from_actor`
- `to_actor`
- `why`
- `scope`
- `expected_output`
- `open_questions`
- `owner_after_return`

## 10. 对当前仓库的落地映射

### 10.1 当前 `synapse` 先保留为内核 id

原型阶段不建议马上删除这些内核 id：

- `overmind`
- `code-expert`
- `research-analyst`
- `finance-scout`

但要把它们的意义收回：

- 这些是内核能力标签
- 不是最终前台人格名称

### 10.2 前台新增“人格层”

前台应新增一层映射：

- `虫群主宰`
- `当前活跃子主脑`
- `该子主脑所属谱系`
- `她当前放出的 brood`

而不是直接把 `finance-scout / code-expert` 暴露成前台名字。

### 10.3 细节层再展示内核

细节层可以显示：

- 当前人格背后调用了哪个内核
- 触发了 `solo / swarm / trial / chain` 哪种结构
- 放出了哪些 units

但普通层不要先给这些。

## 11. 实施建议

### Phase 1：术语对齐

目标：

- 把现有文案从“专家工位”收回到“虫群主宰 / 子主脑 / brood”

内容：

- UI 文案重命名
- 角色命名层抽离
- route_reason 改写成人格责任链语言

### Phase 2：按需分化

目标：

- 不再把子主脑当前台常驻工位

内容：

- 增加 `lineage -> instance` 模型
- 任务开始时由虫群主宰判断是否分化
- 分化出的实例进入任务上下文

### Phase 3：冷藏 / 唤醒

目标：

- 让高频子主脑具有连续性，但不前台常驻

内容：

- 增加冷藏态
- 建 warm pool
- 高频任务优先唤醒已有稳定人格实例

### Phase 4：Brood 显性化

目标：

- 让“子主脑带兵”真正可见

内容：

- brood_run 持久化
- unit 执行过程进入细节层
- 前台继续保持人格语义

---

## 12. 最终结论

更符合愿景的架构不是：

- 一排预设专家人格常驻

而是：

- 一个稳定的虫群主宰
- 一组可按需分化的子主脑谱系
- 每位子主脑背后自带 brood / units
- 分化后可回巢、冷藏、再次唤醒

一句话总结：

**虫群主宰不是在“调用几个专家”，而是在“按任务需要分化自己的局部人格，并指挥她们放出各自的虫群身体去完成任务”。**
