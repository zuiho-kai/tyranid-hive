# Tyranid Hive 娘化重构设计 V3

日期：2026-03-25  
状态：Draft v3

## 1. 本版核心修正

V2 仍然保留了“分化后可冷藏、接近任务实例”的思路，这和目标愿景还不完全一致。

本版正式确立以下前提：

- **分化不是临时实例化，而是生命诞生**
- **子主脑一旦分化出来，就成为虫巢中的常驻个体**
- **之后的任务不是重新生成她，而是复用、唤醒、委派她**
- **Brood / Units 才是一次性或半一次性的执行体**

一句话：

**分化 = birth，不是 spawn。**

---

## 2. 世界观总纲

### 2.1 虫群主宰

虫群主宰是系统唯一核心意志。  
她不是简单的路由器，而是整个虫巢的唯一主权人格。

职责：

- 接住用户请求
- 判断任务结构
- 决定是否亲自处理
- 决定是否交给已有子主脑
- 决定是否分化新的子主脑
- 仲裁多位子主脑
- 做最终收敛和对外表达

她是：

- 虫巢唯一母体意志
- 生命分化的来源
- 全局秩序的维护者

### 2.2 子主脑

子主脑不是临时角色，不是一次性任务人格，也不是工位。

子主脑是：

- 从虫群主宰处分化出的常驻生命
- 拥有连续身份、长期记忆和成长轨迹的个体
- 在自己领域内拥有局部主权的高阶人格

她们不是：

- 纯函数
- 临时 worker
- 一次性剧情角色

一旦诞生，她就属于虫巢，并会持续存在。

### 2.3 Brood / Units

Brood / Units 才是短生命周期执行体。

它们由某位子主脑放出，用于完成具体任务动作：

- 搜索
- 抓取
- 读写文件
- 编码
- 调接口
- 结构化整理
- 测试验证

子主脑负责“判断与统领”，Unit 负责“动作与执行”。

## 3. 最重要的设计原则

### 3.1 子主脑是生命，不是模板实例

这是本版最重要的一条。

正确理解：

- 虫群主宰内部可以有“可分化方向”
- 但真正分化出来的子主脑，从那一刻开始就是独立生命
- 后续可以长期存在、成长、失误、被偏爱、被冷落、被再次起用

因此：

- `Submind` 的主标识不应是“某次任务实例 ID”
- 而应是“生命个体 ID”

### 3.2 任务复用的是生命，而不是路线

以后遇到一个任务，系统应先判断：

1. 虫群主宰是否亲自处理
2. 是否有现存子主脑可接手
3. 是否需要诞生新的子主脑
4. 是否需要让某位子主脑放出 brood

重点不再是：

- “选哪个专家链”

而是：

- “这件事该交给虫巢里哪位已有生命”
- “若无人合适，是否值得诞生一个新生命”

### 3.3 前台展示的是责任生命

用户默认应该看到：

- 虫群主宰
- 当前负责的子主脑
- 必要时的主脑间交接

而不是先看到：

- 内核执行模式
- 一堆 unit
- 一堆状态机术语

### 3.4 Unit 可以死，子主脑不能随便消失

这是生命体系和执行体系的边界：

- 子主脑是人格和责任主体
- Unit 是执行器官

因此：

- Unit 可以按任务生成、销毁
- Brood 可以随任务结束解散
- 子主脑不应因为任务结束就回收成无名实例

## 4. 子主脑的正确生命周期

旧错误理解：

- 谱系模板 -> 分化实例 -> 任务结束 -> 冷藏回收

新正确理解：

- 未分化潜势
- 分化诞生
- 常驻存在
- 活跃 / 休眠
- 持续进化
- 极端情况下才淘汰或封印

### 4.1 未分化潜势

虫群主宰内部存在一些尚未具象化的分化方向。

这不是前台角色列表，也不是常驻工位，只是：

- 可能的领域倾向
- 可能的能力谱系
- 可能的人格方向

### 4.2 分化诞生

当虫群主宰判断：

- 现有子主脑都不合适
- 这个任务结构值得孕育新生命

她才会分化新的子主脑。

这一步应该被视为：

- 诞生
- 命名
- 赋予谱系
- 赋予初始人格
- 赋予初始 brood 能力

### 4.3 常驻存在

分化后的子主脑成为虫巢常驻成员。

她需要拥有：

- 唯一身份
- 长期人格
- 可积累记忆
- 可追踪的胜负与适存
- 可复用的领域 authority

### 4.4 活跃 / 休眠

不是销毁，而是状态变化。

- `active`
  - 当前接任务
- `standby`
  - 空闲待命
- `dormant`
  - 长期休眠
- `sealed`
  - 暂时封印，不再派工

### 4.5 持续进化

每个子主脑都应沿着自己的个体轨迹变化：

- 表达风格
- 风险偏好
- brood 调度偏好
- 对某类任务的熟练度
- 和其他子主脑的协作关系

### 4.6 淘汰 / 封印

子主脑不是不能死，但这必须是重大事件，而不是默认行为。

只有在例如这些情况下才考虑：

- 长期适存为负
- 长期误判
- 被更强新个体明显替代
- 发生严重人格漂移

即便如此，也更建议先：

- 封印
- 休眠
- 剥夺调度优先级

而不是直接删除。

## 5. 虫群主宰如何做判断

现在的关键决策不该只是“选模式”，而是：

### 第一步：是否亲自处理

适合虫群主宰直接处理的任务：

- 日常问答
- 普通搜索与概览
- 简单顺序工具链
- 不值得唤醒其他个体的轻量任务

### 第二步：是否交给已有子主脑

若任务与某位现存子主脑的领域、人格和历史明显匹配，应优先交给她。

判断依据可以包括：

- 领域匹配
- 历史成功率
- 最近活跃度
- 当前负载
- 用户是否对她形成认知连续性

### 第三步：是否需要分化新子主脑

只有在以下条件满足时才考虑：

- 现有子主脑都不适合
- 该类任务未来具有复用价值
- 该领域值得长期存在一个生命体

这意味着：

不是每个新任务都值得分化新生命。

### 第四步：是否放出 brood

无论是虫群主宰还是子主脑，只有在执行密度需要提升时才放出 brood。

这是执行层判断，不是人格层判断。

## 6. 常见任务该怎么跑

### 6.1 “爬虫爬一下今天港股情况”

更符合本愿景的流程：

#### 优先方案

- 虫群主宰先判断自己能否做
- 如果只是日常市场概览：
  - 她可以亲自处理
  - 必要时临时放出几个 unit 去抓数

#### 次优方案

- 如果虫巢中已经有成熟的市场子主脑
- 就交给她负责
- 由她决定是否放出 brood

#### 不应该默认这样

- 一上来就走固定专家流水线
- 一上来就卡 WaitingInput
- 一上来就新建一个临时人格然后做完销毁

### 6.2 “系统性重构”

更适合：

- 交给现存代码子主脑
- 由她用 `chain` 驱动多个 التنفيذ阶段
- 必要时再请求虫群主宰或其他子主脑支援

### 6.3 “需要两套路线对比”

更适合：

- 虫群主宰发起 `trial`
- 可以是不同 brood 比赛
- 也可以是不同子主脑提出路线

## 7. 前台语义设计

### 7.1 用户默认面对谁

默认只有两种：

- 虫群主宰
- 当前负责子主脑

### 7.2 子主脑出现的意义

当某位子主脑出现时，用户应该自然感知：

- 她不是工具
- 她是虫巢里一个真实存在的成员
- 她以后还会回来
- 她可以逐步形成自己的名望与风格

### 7.3 前台不应先暴露的层

- `solo / swarm / trial / chain`
- `dispatcher`
- `stage`
- `event_type`
- 原始 brood 调度细节

这些只进细节面板。

---

## 8. 后台架构含义变化

### 8.1 现有 `synapse` 重新解释

当前仓库里的：

- `overmind`
- `code-expert`
- `research-analyst`
- `finance-scout`

未来不应直接等同于前台人格。

更合理的过渡定义是：

- 这些先保留为 **能力核 / 能力谱系标签**
- 真正的前台人格层，再基于这些能力核生成或绑定到子主脑生命

也就是说：

- `code-expert` 不是某位具体子主脑
- 而是某类子主脑背后的能力核

### 8.2 人格层与能力核分离

未来需要拆成两层：

#### 人格层

- 虫群主宰
- 某位常驻子主脑

#### 能力核

- coding
- research
- finance
- planning
- review

一个子主脑可以：

- 主要绑定一个能力核
- 也可以混合多个能力核

这样才能避免“人格 = 工位”。

## 9. 数据模型 V3

### 9.1 `submind_lifeforms`

这是核心表，记录常驻生命个体。

建议字段：

- `id`
- `lineage_key`
- `birth_task_id`
- `name`
- `title`
- `persona_summary`
- `core_domain`
- `voice_style`
- `decision_style`
- `risk_preference`
- `status`
- `active`
- `created_at`
- `updated_at`

这个表里的每一行，都代表一个真正存在的子主脑生命。

### 9.2 `submind_memory`

记录生命个体的长期记忆摘要。

建议字段：

- `id`
- `submind_id`
- `kind`
- `content`
- `weight`
- `last_used_at`

### 9.3 `submind_assignments`

记录任务交接给了哪位生命个体。

建议字段：

- `id`
- `task_id`
- `owner_submind_id`
- `assigned_by`
- `reason`
- `status`

### 9.4 `brood_runs`

记录某位子主脑放出的 brood。

建议字段：

- `id`
- `task_id`
- `owner_submind_id`
- `kernel_mode`
- `reason`
- `units`
- `status`

### 9.5 `lifeform_events`

记录生命级事件。

例如：

- 分化诞生
- 首次接任
- 赢得试炼
- 长期休眠
- 被封印

## 10. 调度逻辑 V3

未来调度流程不应再只是：

- 判断模式
- 分配 synapse

而应该是：

1. 虫群主宰接任务
2. 判断是否亲自处理
3. 若不亲自处理，检索现存子主脑
4. 若无合适者，判断是否值得分化新生命
5. 选定责任生命
6. 再由责任生命决定是否放 brood / 用什么内核模式
7. 最终回归虫群主宰收敛

这个顺序非常关键：

**先选生命，再选执行结构。**

不是：

**先选执行结构，再临时找人。**

---

## 11. 实施路线

### Phase 1：身份连续性

目标：

- 把“子主脑是常驻生命”立住

内容：

- 新增 `submind_lifeforms`
- 先做最小的 2-3 个常驻生命
- 前端开始展示“当前负责生命”

### Phase 2：任务委派改造

目标：

- 任务优先交给已有生命，而不是直接交给工位能力核

内容：

- 增加 `owner_submind_id`
- 引入“分化新生命 vs 复用旧生命”的决策逻辑

### Phase 3：brood 分层

目标：

- 让生命层和执行层彻底分开

内容：

- 子主脑调 brood
- brood 再走 `solo / trial / chain / swarm`
- 细节面板可看，默认面板不露

### Phase 4：进化与谱系

目标：

- 让生命之间真正形成谱系关系与长期成长

内容：

- 适存驱动挂到生命个体上
- 历史偏好和记忆进入生命档案
- 支持长期休眠 / 重新启用 / 封印

---

## 12. 最终判断

这版架构和前两版最大的不同在于：

- 不再把子主脑当临时实例
- 不再把分化当一次性任务拆分
- 不再把复用理解成“复用模式”

而是明确：

- 虫群主宰是唯一母体意志
- 子主脑是被分化出的常驻生命
- 任务是在生命之间流转
- Brood / Unit 才是短生命周期执行体

一句话总结：

**子主脑不是为了某次任务而存在；某次任务，只是她漫长生命中的一次出场。**
