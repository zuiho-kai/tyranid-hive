# Tyranid Hive 娘化重构设计

日期：2026-03-25  
状态：Draft

## 1. 目标

当前项目已经有虫群执行核，但前台体验仍然更像“状态机 + 调度器”，不像一个有角色、有关系、有责任归属的活体系统。

本次重构的目标，不是简单套用“三猫模板”，而是把 **Tyranid Hive 做成“娘化虫群”**：

- `Overmind` 是系统唯一总主脑，类似“管家婆正宫”
- 其他 `Submind` 是各自领域的“子主脑 / 子女主”
- 每个子主脑都有：
  - 自己的专精领域
  - 自己的人设、表达风格、判断偏好
  - 自己统领的 `Brood / Units`
- 用户主要和“主脑人格”对话，不直接面对底层执行单位
- 底层仍然保留 `solo / trial / chain / swarm` 的虫群执行核

一句话：**前台是娘化主脑生态，后台是虫群执行系统。**

---

## 2. 愿景判断

### 2.1 当前问题

当前系统的主要偏差：

- `Overmind` 更像调度器，不像“有责任感的总主脑”
- `Submind` 更像专家工位，不像有主见的域内主脑
- `Chain` 容易退化成“专家流水线”
- UI 暴露了太多内部状态和阶段词汇
- 用户知道“系统在跑”，但不知道“是谁在负责”

### 2.2 重构后理想状态

用户的心理感受应该变成：

- 我在和一个“虫巢后宫”里的角色说话
- 这位角色有稳定人格，不是随机函数
- 她会自己判断要不要叫别人
- 她背后会调度自己的虫群去干活
- 最终一定有人对结果负责，而不是状态机自己跳来跳去

## 3. 核心世界观

### 3.1 角色层级

#### Overmind

系统唯一总主脑。  
人格定位：稳、强、能控局、能收敛。  
叙事比喻：`管家婆正宫`。

职责：

- 接住用户请求
- 判断由谁负责
- 决定是否直接处理，还是委派子主脑
- 仲裁多路意见
- 给用户最终收敛结果

她不应该：

- 退化成只输出 JSON 的路由器
- 对每个任务都亲自下场执行
- 把内部状态机直接甩给用户

#### Submind

每个子主脑都是一个拟人化角色。  
她不是“专家工具人”，而是“域内 CEO”。

每个 Submind 都有：

- 领域专精
- 拟人化设定
- 固定语气
- 偏好与判断倾向
- 自己的 `Brood / Units`
- 和其他 Submind 的协作关系

#### Brood / Units

这是主脑的身体，而不是用户交互对象。

它们负责：

- 搜索
- 抓取
- 读文件
- 跑代码
- 调接口
- 结构化处理
- 验证结果

用户默认不需要直接看到 Unit，除非主动展开细节。

## 4. 角色设计原则

### 4.1 角色必须拟人，但不能空心

角色设计不能只停留在名字和语气。

每个主脑必须绑定：

- `领域`
- `人格`
- `方法论`
- `判断偏好`
- `可调用 unit 集`
- `可向谁发起 handoff`

### 4.2 角色不是工位

错误做法：

- 金融虫负责抓取
- 研究虫负责汇总
- 代码虫负责写代码

这种结构太像流水线专家工位。

正确做法：

- 金融子主脑可以自己先判断市场任务怎么做
- 她可以自己做，也可以叫自己的 unit 并行抓取
- 她也可以把一部分工作交给别的主脑
- 最终她仍然是该任务的域内负责人

### 4.3 默认归责必须清晰

任何时刻，系统里必须有一个明确的“当前负责主脑”。

用户应该能一眼看出：

- 现在谁在负责
- 为什么是她负责
- 她是否把任务交给了别人
- 最后谁来收敛答案

## 5. 推荐角色体系

下面先给一版原型期可落地的最小角色集。

### 5.1 Overmind

- 对外称呼：`总主脑`
- 人设关键词：沉稳、收束、掌控全局
- 主要职责：接任务、裁决、收敛、对用户负责

### 5.2 代码子主脑

- 暂定代号：`玄猫`
- 领域：编码、重构、调试、工程实现
- 人设关键词：冷静、干练、执行力强
- 背后 unit：
  - 代码检索 unit
  - 修改实现 unit
  - 测试验证 unit

### 5.3 研究子主脑

- 暂定代号：`白猫`
- 领域：信息收集、对比分析、资料整理
- 人设关键词：清醒、条理、擅长总结
- 背后 unit：
  - 搜索采集 unit
  - 来源校验 unit
  - 摘要整理 unit

### 5.4 金融子主脑

- 暂定代号：`金猫`
- 领域：行情、市场、财务数据、资金面
- 人设关键词：敏锐、势利、数字感强
- 背后 unit：
  - 指数抓取 unit
  - 板块/个股抓取 unit
  - 资金流分析 unit

### 5.5 挑战 / 审校子主脑

- 暂定代号：`影猫`
- 领域：反证、风险、审查、Review
- 人设关键词：挑剔、警觉、专门拆错
- 背后 unit：
  - 风险清单 unit
  - 反例检索 unit
  - 验收核查 unit

## 6. 执行哲学

## 6.1 前台语义

前台始终是：

- 用户
- 总主脑
- 某位当前负责的子主脑

允许出现 handoff，但不默认把底层模式词暴露给用户。

用户看到的是：

- “这件事我来处理”
- “我叫谁来帮我”
- “为什么叫她”
- “她做完后我来给你结果”

### 6.2 后台执行

后台仍使用虫群核：

- `solo`
- `trial`
- `chain`
- `swarm`

但这些是“内核模式”，不是“前台人格”。

举例：

用户说：`爬虫爬一下今天港股情况`

前台：

- 总主脑：这件事交给金融子主脑
- 金融子主脑：我先做今天港股概览，必要时我会让下面的单位并行取数
- 金融子主脑：这是结果

后台可能是：

- `solo`：金融子主脑顺序工具链完成
- `swarm`：金融子主脑放出 3 个 unit 分别抓指数 / 资金流 / 活跃个股
- `trial`：两种数据源或两种抓取策略赛马

### 6.3 模式约束

- `Solo`
  - 默认模式
  - 日常任务优先
  - 由当前负责主脑直接完成

- `Swarm`
  - 当任务可拆成多个彼此独立的采集或执行单元时触发
  - 属于“主脑调自己的兵”

- `Chain`
  - 只用于严格线性依赖任务
  - 例如系统重构、阶段性产物必须串行接棒
  - 不用于“专家流水线”

- `Trial`
  - 当存在多条独立可行路径，且结果需要比较时触发
  - 应由总主脑或当前负责主脑发起

## 7. Handoff 设计

这是当前系统最缺的部分，也是重构关键。

Handoff 必须成为一等对象。

每次交接至少包含：

- `from`
- `to`
- `why`
- `scope`
- `expected_output`
- `open_questions`
- `owner_after_return`

例如：

```json
{
  "from": "overmind",
  "to": "finance-submind",
  "why": "这是金融市场概览任务，应由金融子主脑负责",
  "scope": "完成今日港股默认概览",
  "expected_output": "可直接给用户的市场摘要",
  "open_questions": [],
  "owner_after_return": "overmind"
}
```

这会直接决定：

- UI 怎么显示交接卡
- 细节面板怎么解释“为什么是她”
- 事件日志怎么记录

## 8. 前端设计

### 8.1 默认界面

左侧：

- 线程列表
- 当前负责主脑标识

中间：

- 主线程
- 用户消息
- 总主脑 / 子主脑消息
- handoff 卡片

右侧：

- 默认折叠
- 只在需要时展开：
  - 为什么这样路由
  - 当前负责者
  - handoff 记录
  - brood / unit 明细
  - 原始事件

### 8.2 UI 原则

- 普通用户默认只看“谁在负责”
- 不先看 `trial / chain / swarm`
- 不先看 `dispatcher / stage / task.state`
- 默认优先展示人格和责任链

### 8.3 细节层

当用户点开细节时，再展示：

- 内核执行模式
- 触发原因
- brood 列表
- unit 执行过程
- 原始事件账本

## 9. 后端设计

### 9.1 新增概念

建议新增三类实体：

#### `submind_profiles`

描述每只子主脑的人设和能力。

字段建议：

- `id`
- `name`
- `title`
- `persona_summary`
- `domain`
- `voice_style`
- `decision_style`
- `default_visibility`
- `active`

#### `handoffs`

描述主脑间交接。

字段建议：

- `id`
- `task_id`
- `from_actor`
- `to_actor`
- `why`
- `scope`
- `expected_output`
- `open_questions`
- `status`
- `created_at`

#### `brood_runs`

描述某位主脑在后台放出的虫群执行。

字段建议：

- `id`
- `task_id`
- `owner_submind`
- `kernel_mode`
- `reason`
- `units`
- `status`
- `created_at`

### 9.2 兼容现有内核

现有这些模块可以保留为虫群核：

- `mode_router.py`
- `dispatcher.py`
- `trial_race.py`
- `chain_runner.py`
- `swarm_runner.py`

但它们不应直接决定前台角色语义。  
前台语义应由：

- `Overmind`
- `Submind profile`
- `handoff`
- `brood_run`

共同决定。

## 10. 当前仓库映射建议

### 10.1 现有 synapse 命名先不立刻删

原型期不建议马上推翻：

- `overmind`
- `code-expert`
- `research-analyst`
- `finance-scout`

更务实的方式是：

- 保留内核 id
- 新增前台人格映射

例如：

- `overmind` → `总主脑`
- `code-expert` → `玄猫`
- `research-analyst` → `白猫`
- `finance-scout` → `金猫`

### 10.2 UI 先做“人格映射层”

前端先把现有 `synapse` 映射成“主脑角色卡”。

这样可以不动底层执行引擎，先把外部体验改对。

### 10.3 路由理由改成角色语义

现在的 `route_reason` 多数还是技术型描述。  
后续应该改成人格责任描述，例如：

- `总主脑判断这是市场任务，先交给金猫负责`
- `金猫决定放出 3 个单位分别采集指数、资金流和活跃个股`
- `影猫介入复查，补充了风险提示`

## 11. 分阶段实施计划

### Phase 1：人格壳

目标：先把“谁在负责”做出来。

内容：

- 新增 `submind profile` 映射层
- UI 中用人格名替代工位名
- DetailPanel 展示“当前负责主脑”
- handoff 先以轻量 meta 形式落地

交付标准：

- 用户能看懂是谁在接任务
- 用户能看懂为什么转给另一位主脑

### Phase 2：交接链

目标：把 handoff 变成真实对象。

内容：

- 新增 `handoffs` 表
- 前端展示 handoff 卡
- 总主脑和子主脑消息分层展示
- 责任归属清晰可追踪

交付标准：

- 每一次交接都可追溯
- 用户能看懂“谁交给谁”“为什么”

### Phase 3：Brood 显性化

目标：让“主脑带兵”真正成立。

内容：

- 新增 `brood_runs`
- swarm / trial / chain 成为主脑背后的 brood 动作
- 细节层可展开 brood / unit
- 默认层仍保持简洁

交付标准：

- 前台是人格协作
- 后台是虫群执行
- 两层语义不打架

---

## 12. 最终判断

这次重构的关键不是“把名字换成猫”，而是把系统从：

- 调度器驱动的多 Agent 面板

改成：

- **有责任主体、有角色关系、有 brood 执行体的拟人化虫巢**

最终理想形态：

- Overmind 像正宫，能定局
- Submind 像各房主脑，各自有性格、有擅长领域
- Brood / Units 是她们的身体和兵器
- 用户看到的是角色生态
- 系统内部跑的是虫群核

一句话总结：

**不是把泰伦虫群“换皮成猫”，而是把虫群的组织结构，翻译成一个能被人直觉理解、又足够有魅力的娘化主脑生态。**
