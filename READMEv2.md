## 项目定位

`Tyranid Hive` 是一个受泰伦虫群意象启发的任务编排原型。

它想做的，不是普通的多 Agent 面板，也不是把“三猫协作”直接换皮，而是一套“前台拟人化虫群、后台虫群执行核”的系统：

- 对外：用户面对的是有生命感、可承担责任的虫群人格
- 对内：系统保留 `solo / trial / chain / swarm` 等执行结构

当前冻结下来的核心语义：

- `虫群主宰`
  - 系统唯一核心意志
  - 负责接住任务、判断结构、决定是否亲自处理或交接、最后收敛结果
- `子主脑`
  - 从虫群主宰处分化出来的常驻生命体
  - 不是一次性 worker，不是临时专家工位
- `brood`
  - 某位生命体围绕当前任务放出的执行群组
- `unit`
  - brood 内的最小执行单元

一句话：

> 前台是拟人化虫群，后台是虫群执行核。

## 为什么不是普通多 Agent

这个项目当前明确反对几种退化方向：

- 退化成“专家流水线”
  - 例如固定成 “A 负责采集，B 负责总结，C 负责 review”
- 退化成“状态机前台”
  - 用户主要只看到 `Planning -> Reviewing -> Spawning`
- 退化成“角色模板换皮”
  - 只是把别人的角色设定换成虫群术语

当前更接近的目标是：

- 用户默认先看到责任链，而不是先看到执行模式
- 生命体负责“谁在负责、为什么交接、谁来收敛”
- 执行核负责“具体怎么跑”

## 当前项目状态

截至 2026-03-26，这个仓库已经不是纯愿景文档，而是一个可运行原型。

已经落地的部分：

  - `Lifeform`
  - `Assignment`
  - `Handoff`
  - `Task.current_owner_lifeform_id`
  - `Task.entry_lifeform_id`
  - `Task.last_handoff_id`

还在推进中的部分：

- 前端仍在从“状态机 UI”切到“责任人格链 UI”

  - 分化
  - 休眠
  - 唤醒
  - 封印
  - 结晶
- `brood / unit` 还主要借现有执行层和事件流承载，没有完全独立建模
- 元治理规则已经写入文档，但尚未进入代码实现

一句话总结现状：

- 愿景已经明确
- 原型已经能跑
- 语义重构正在进行
- 还远没到成品阶段

## 当前重构方向

这轮重构的重点不是“再加几个模式”，而是把系统表达改对。

目标是把项目从这套表达：

- 状态机
- 执行模式
- 专家工位

转成这套表达：

- 谁先接住任务
- 现在谁在负责
- 为什么交给她
- 她下面放了哪些执行体
- 最后由谁来收敛

当前重构原则：

1. 默认先看责任链，不先看内部状态机
2. 默认先看生命体，不先看执行核
3. 简单任务优先由虫群主宰直接处理
4. 不把系统退化成专家流水线
5. 不把每个新任务都当作必须诞生新生命

## 当前 UI 语义

Dashboard 现在正在往“责任人格链”视角收。

目标界面分三层：

- 左侧：任务列表
  - 只看任务、负责人、基本状态
- 中间：任务主线
  - 只看用户输入、虫群主宰接球、责任交接、关键进展
- 右侧：细节抽屉
  - 按需查看路由原因、分化结构、执行过程、原始事件

也就是说：

- 默认层是“谁在负责”
- 细节层才是“内部怎么跑”

### 当前已经做到的 UI 变化

- 左侧任务卡会显示负责人
- 中间主线开始显示：
  - 用户输入
  - 虫群主宰接球
  - 责任交接
  - 执行进展
- 右侧细节开始显示：
  - 当前负责人
  - 交接原因
  - 分化结构
  - 执行过程
  - 原始事件

### 还没完全收口的部分

- 仍有一些旧组件沿用旧术语
- 历史任务数据和旧文案可能会夹杂旧表达
- 还没有完整把“主宰 / 子主脑 / brood”的关系讲到足够顺

## 执行核现状

当前底层仍保留原来的执行核结构。

已经打通或部分打通的能力：

- `solo`
- `trial`
- `chain`
- `swarm`

但它们在当前项目里的定位应该是：

- 这是后端执行结构
- 不是默认前台语言

也就是说，普通用户不该先看到：

- 这是不是 chain
- 这是不是 swarm

普通用户应该先看到：

- 现在谁在负责
- 为什么换人
- 结果什么时候回来

## 当前适合做什么

这版更适合：

- 本地任务编排原型验证
- 责任链 / handoff / 路由语义验证
- Windows + Codex 执行链联调
- UI 语义和愿景的快速迭代
- 文档冻结和架构收敛

这版还不适合：

- 生产级长任务托管
- 完整自动进化
- 大规模 brood / unit 生命周期治理
- 多 Hive 协作
- 完整的自治生命群管理

## 当前已做的关键修正

最近几轮里，已经做过一些关键修正：

- 修掉了 Windows 下 `codex.CMD` 的长参数导致的执行失败

这些修正不是最终形态，但至少把系统从“只能跑”往“开始讲得清楚”推了一步。

## 接下来优先级

当前 Phase 1 的重点不是“做更多模式”，而是把责任人格链先立稳。

优先级如下：

1. 前端彻底改成责任链语义
2. 让 handoff 成为一等数据，不只停在 API 和事件层
3. 把 `虫群主宰 -> 子主脑 -> brood` 的关系在界面里讲清楚
4. 再推进真正的分化、唤醒、封印和结晶

Phase 1 之后，才值得继续考虑：

- 更完整的子主脑生命周期
- brood / unit 的独立建模
- 元治理进入实现
- 更可靠的长期任务承接
